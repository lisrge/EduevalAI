# 项目博客摘要总汇（lsz / lzc / pzl）

## lsz 工作总结（前端）

lsz 负责前端工程搭建与主流程页面落地，目标是把“登录 → 申请书上传/预览/评分 → 个人文档管理 → 管理后台入口”串成可用闭环。

- 技术理解：用 Vue3 组件化拆分页面职责；用 Pinia 统一承载系统级状态（登录态、列表态、队列状态）避免跨页面传参；用 Vue Router 的 meta + 全局守卫收口鉴权逻辑，保证权限规则一致且可扩展。
- 技术进度：已完成登录/注册与登录态恢复；申请书工作台（列表、上传、预览、评分入口）；我的文档（申请书/任务书草稿与编辑器）；个人空间；后台管理入口与数据管理表格页面；预留聊天与拖拽上传入口。
- 关键实现：批量上传采用“队列 + 有限并发”，并用状态机回写 UI（排队/进行中/成功/失败），把高并发不确定性转成可控流程。

**草稿 vs 归档（本项目中的区别）**
- 草稿：面向“编辑过程”，支持高频修改与自动保存，内容更偏结构化/可编辑，不进入后端的抽取/评分/统计主链路。
- 归档：面向“最终提交”，将草稿导出为最终文件并上传，进入后端的抽取、预览、评分与导出等处理链路，强调可追溯与可统计。

**草稿 vs 归档（本项目中的存储方式）**
- 草稿怎么存：
  - 存在数据库表 `application_drafts` / `task_drafts`，核心内容存为 `content_json`（JSON 字符串），并带有标题、状态、创建/更新时间与所属用户等元数据。
  - 导出草稿为 DOCX 时是按需生成并直接返回下载流，不把“草稿文件”长期落盘。
- 归档怎么存：
  - 文件本体上传后写入后端本地目录 `backend/storage/applications/`（文件名带 UUID 前缀防重名），数据库表 `application_records` 记录 `file_path/file_name/file_type/student_id` 等信息，以及抽取/评分状态。
  - 预览相关缓存（例如 DOCX 转 HTML）会落在 `backend/storage/previews/` 以便复用与加速。

### lsz 博客简单摘要

- 1：前端工程入口与路由/登录态的基础搭建，强调启动顺序、懒加载与统一鉴权。
- 2：申请书模块的组件拆分与关键交互，重点是队列上传、单项/批量评分与状态回写。
- 3：从架构角度总结前端底座：启动顺序、系统级状态边界、访问控制收口位置。
- 4：提出“草稿—归档”双通道：草稿用于频繁编辑，归档用于触发后端处理链路，提升维护性与体验。

## lzc 工作总结（后端基础设施 + 博客系统）

lzc 负责后端框架基础设施与数据层能力建设，并实现独立的“工作日志/博客”模块与管理员管理能力，确保后端具备可持续迭代的工程底座。

- 技术理解：用 FastAPI 模块化路由按领域拆分接口；用 Pydantic Settings 统一管理环境配置，避免硬编码；用 SQLAlchemy ORM + Session 让数据库访问以“请求内工作单元”形式可控提交与回滚。
- 技术进度：已完成后端入口与跨域配置；数据库连接、建表与核心模型/Schema 约束；用户与会话分离建模；迷你博客系统的发布/存储/渲染链路；管理员侧查看、删除、搜索与权限校验等管理闭环。
- 关键实现：权限初始化采用“首个用户自动成为管理员”降低部署门槛；博客展示采用“Markdown 原文 + HTML 缓存”减少重复转换；敏感操作通过“接口语义化（DELETE）+ 权限判定”保证边界清晰。

### lzc 博客简单摘要

- 1：后端服务入口与配置/数据库底座搭建（CORS、路由注册、Settings、Session 管理）。
- 2：用户与会话模型的拆分与安全存储设计（密码哈希、token 哈希、关联关系与级联清理）。
- 3：迷你博客系统的技术选型与核心流程（MD 存储、MD→HTML、作者关联、基础认证）。
- 4：管理员后台能力与权限校验（自动管理员、删除/搜索/退出等管理功能的接口与边界）。

## pzl 工作总结（后端核心业务：评分/预览/导出/爬虫接口契约）

pzl 负责后端核心业务能力与可用性保障，围绕“文件进入系统后如何被处理并产出可展示、可导出的结果”展开。

- 技术理解：用“路由层（接口契约）—业务层（服务实现）”分层降低耦合；对模型服务等外部依赖用降级机制控制不确定性；对输出坚持结构化约束以便落库与展示。
- 技术进度：已完成文本提取（PDF/DOCX/纯文本）与安全落盘；登录会话与 token 校验；AI 评分（模型调用 + 规则回退）；预览（PDF 原生、DOCX 转 HTML + 缓存）；评分结果导出（CSV/Excel）；同时完成“学号 ↔ 博客入口地址”绑定数据的落库与管理端查看能力，为爬虫提供稳定入口；并在接口与数据模型层预留爬虫的批量写入、幂等更新与判定状态输出的契约，便于后续接入真实爬虫任务。
- 关键实现：评分侧以结构化 JSON 输出为契约；不可用时回退规则评分保证服务可用；预览侧用统一入口屏蔽格式差异并缓存复用；导出侧内存生成文件降低 IO 成本。

### pzl 博客简单摘要

- 1：底层能力：文本提取、文件安全保存、认证与会话管理的基础实现与安全边界。
- 2：核心业务：AI 评分（提示词 + 结构化输出 + 回退）、预览（DOCX→HTML 缓存）、导出（CSV/Excel）。
- 3：爬虫协作：BlogPost 与绑定表的设计，明确“学号 → 博客入口地址”作为爬取起点；写入侧建议按 url 做幂等 upsert 并支持批量提交，方便重复跑任务与增量更新；鉴权与用户 token 体系隔离，避免爬虫持有管理员权限；同时补充失败重试、限流、去重、清洗归一化与日志追踪等稳定性建议。
- 4：博客判定：normal/abnormal 的后端流水线设计（规则基线、可解释 reason、版本追溯与复核边界）。

## 开发过程中的技术选择问题（示例）

1. 一开始使用 SQLite，后更换为 MySQL 的原因

- 并发与多用户：SQLite 以文件锁为主，写入并发高时容易出现锁等待；MySQL 对并发写入与多连接更友好。
- 多人协作与部署：SQLite 文件难以在多实例/多容器场景共享；MySQL 支持独立数据库服务，便于后端水平扩展与统一管理。
- 运维与能力：MySQL 在备份恢复、权限管理、连接池、监控与审计方面更成熟，适合后期稳定运行。
- 数据演进：随着表与查询变复杂，MySQL 在索引、查询优化与事务能力上更易支撑长期迭代。

1. 其他技术选择问题

- 文件存储：本地磁盘 vs 对象存储（MinIO/OSS）
  - 我们当前做法：文件先落盘到后端本地目录，并记录路径与元数据到数据库，预览/评分/导出都以此为输入。
  - 相对优点：开发成本低、链路短，便于快速把“上传—预览—评分”闭环跑通；调试时文件可直接定位；对小规模使用场景足够。
  - 改进点：当后端多实例部署或需要高可用时，本地磁盘难以共享；可迁移到对象存储并引入签名下载/上传、生命周期管理与统一备份，提升可扩展与可靠性。
- 预览方案：DOCX 转 HTML vs 统一转 PDF
  - 我们当前做法：预览文件不在上传时生成，而是在用户点击预览（请求 `/preview`）时按需生成：PDF 直接返回原文件；DOCX 首次预览时后端转换为 HTML 并写入 `storage/previews/`，若源文件未变化则复用缓存。
  - 相对优点：HTML 方案转换速度快、易注入主题样式；与前端 iframe 预览融合度高；按需生成避免上传时额外耗时；缓存后重复打开成本低。
  - 改进点：不同 DOCX 的排版兼容性存在差异；若追求“渲染一致性”，可增加统一转 PDF 的可选路径（例如用于最终归档/打印），并在性能与一致性之间提供配置开关。

# 后端：博客“正常/不正常”判定流程设计

本文讨论博客质量判定（normal/abnormal）的后端设计思路。当前项目对“不正常”的定义包括两类：  

- 只是粘贴代码的  
- 只是进行科普的  

目标是让后端输出一个稳定的 `status`，供前端统计展示与后续的人工复核。

## 1. 为什么要把判定放到后端

判定逻辑通常涉及：

- 文本特征抽取（代码占比、标题/正文结构、链接密度等）
- 可能的模型推理（分类模型、LLM 判别）
- 版本迭代与可追溯（同一篇博客在不同版本规则下结果不同）

这些都不适合放前端：

- 前端不应承载不可信推理与复杂计算
- 前端升级不可控，逻辑容易分裂
- 判定规则属于业务核心，应可控、可迭代、可回放

## 2. 判定输入与输出

### 2.1 输入

建议判定服务接收：

- `url`：博客链接
- `content_md`：清洗后的 Markdown
- 可选：原始 HTML、作者信息、发布时间、标签等

### 2.2 输出

最小输出：

- `status`: `normal` 或 `abnormal`
- `reason`: 判定理由（用于后台复核/解释）

进阶输出（推荐后续加）：

- `score`: 置信度或质量评分（0-1 或 0-100）
- `labels`: 多标签（例如 `code_only`、`popular_science_only`、`ad_heavy`）
- `rule_version` / `model_version`: 方便回溯

## 3. 规则优先：先落地一个可解释的基线

在没有训练好的分类模型之前，建议先用规则做一个“可解释基线”，例如：

### 3.1 “只是粘贴代码”检测

可用特征：

- 代码块行数 / 总行数占比过高
- 连续长行、符号密度高（`{}`, `();`, `< >` 等）
- 自然语言段落很少（段落数、句子数低）

简单阈值示例（仅示意）：

- 代码占比 > 70% 且自然语言段落 < 3 => `abnormal: code_only`

### 3.2 “只是科普”检测

科普并非一定不好，但如果项目要求“必须是项目实践/过程/结果”，那么“只讲概念、无项目落地”可标为不正常。

可用特征：

- 缺少项目要素：问题背景、实现过程、数据/截图、对比实验、复盘等
- 多为定义/概念解释，缺少第一人称实践描述

规则可以非常保守：先把疑似科普标为 `needs_review`（如果未来扩展 status 枚举），或直接 `abnormal` 并记录 reason 供管理员复核。

## 4. 推荐的判定流水线（Pipeline）

一个可迭代的后端流水线可以是：

1) 爬虫抓取并清洗 -> 得到 `content_md`
2) 提取特征 -> code_ratio、paragraph_count、keyword_hits 等
3) 规则判定（baseline）
4) （可选）模型判定覆盖/校正
5) 写入 `BlogPost.status`，同时写入 `reason/model_version`

目前数据库里 `BlogPost` 已有 `status` 字段，可先落地第 1/2/3/5 步。

## 5. 与管理后台的联动

前端后台用户列表展示正常/不正常的计数，本质上依赖：

- `status` 值稳定（例如固定为 `normal`、`abnormal`）
- 后端按 `user_id + status` 聚合查询

为了后续更细化，建议后端内部可以保留更详细的 labels/reason，但对前端暴露仍归一为两类：

- normal
- abnormal

## 6. 风险与边界

- 内容很短的博客：容易误判，建议加入“最小长度阈值”，低于阈值先标记为 abnormal 或待人工复核
- 混合型博客：既有大量代码也有大量解释，规则阈值应谨慎，宁可多标记“需复核”
- 抄袭/搬运：需要额外的相似度检测（后续再做）

## 7. 小结

后端判定的关键不是一开始就“判得很准”，而是：

- 有稳定输出（status）
- 有可解释依据（reason/labels）
- 有可迭代版本（rule/model version）

这样管理员在后台看到红色计数时，能够追溯原因并逐步提升判定质量。

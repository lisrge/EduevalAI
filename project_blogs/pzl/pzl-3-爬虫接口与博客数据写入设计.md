# 后端：爬虫接口与博客数据写入设计

本文从后端角度说明：当“博客内容由爬虫同学负责抓取”时，本项目应如何设计数据模型与写入接口，让爬虫侧可以稳定入库、前端可以稳定展示、管理员可以在后台统计“正常/不正常”博客数量。

## 1. 核心对象：BlogPost

博客的最小落库单元是 BlogPost（每篇一条记录），通常包含：

- user_id：归属的用户（系统里用用户维度展示/统计）
- title：标题
- url：原始博客链接
- status：normal/abnormal（或更多枚举）
- content_md：抓取并清洗后的 Markdown 正文
- created_at/updated_at：时间戳

这个模型面向“展示与统计”，因此字段尽量以可检索、可展示为目标设计。

## 2. 学号 ↔ 博客地址：StudentBlogBinding

为了“方便后端调用”，项目新增了绑定表，表示：

- 某个学号对应一个博客地址（可理解为博客主页/项目博客地址）
- 来源通常来自任务书上传（表单填写后落库）
- 同一学号重复上传时覆盖更新（upsert）

它解决的问题是：

- 爬虫不必理解复杂的前端表单结构
- 爬虫只需要按学号拿到一个稳定的入口 URL
- 后续爬虫可批量拉取绑定表，做增量更新

## 3. 推荐的爬虫写入方式：幂等 + 批量

爬虫侧常见调用形态有两种：

### 3.1 幂等写入（单篇 upsert）

用 `url` 或 `(user_id, url)` 作为唯一性标识，写入时：

- 如果记录存在：更新 content_md/status/updated_at
- 不存在：新增一条

好处是重复跑任务不会生成重复数据。

### 3.2 批量写入（一次提交多篇）

当一次爬取产生几十/几百篇内容时，建议后端提供批量接口：

- 接收一个列表
- 逐条 upsert
- 返回成功/失败明细与统计计数

批量的好处是减少网络往返，提高吞吐，也便于爬虫失败重试。

## 4. 接口层建议：单独的 crawler 路由

虽然目前前端只要求“留接口”，但从工程实践上建议新增：

- `/api/crawler/bindings`：读取学号 ↔ 博客地址绑定表
- `/api/crawler/blog-posts`：批量写入博客
- `/api/crawler/blog-posts/{id}`：单条更新（可选）

鉴权建议：

- 单独的 crawler token（与用户 token 分离）
- 或者 IP 白名单 + token

避免用管理员 token 直接给爬虫调用，减少权限泄漏风险。

## 5. 清洗与归一化：把不稳定留在爬虫侧

博客源站 HTML 的差异很大，建议爬虫侧完成：

- HTML -> Markdown 的转换
- 去噪（广告、导航、脚本、版权声明等）
- 图片与链接的处理策略（保留/替换为纯文本/下载入库）

后端侧尽量只做：

- 字段校验（url 合法、content_md 不为空等）
- 存储与检索
- 基本的长度限制（防止异常大内容撑爆数据库）

## 6. 与前端的配合点

前端后台用户列表里展示“博客发布篇数”，并按 status 分为：

- normal：绿色计数
- abnormal：红色计数

这意味着后端需要保证：

- status 字段稳定可枚举
- 统计查询能按 user_id + status 聚合

## 7. 小结

面向“爬虫同学 + 管理后台”的博客模块设计要点是：

- BlogPost 面向展示与统计
- StudentBlogBinding 面向爬虫输入源（学号到入口 URL）
- 接口要幂等（支持重跑）、最好支持批量
- 鉴权要与用户权限体系隔离

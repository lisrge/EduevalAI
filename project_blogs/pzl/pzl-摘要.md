# pzl 文件夹博客摘要（1-4）

## 个人工作总结（pzl）

pzl 主要负责后端核心业务能力的实现与可用性保障，围绕“文件进来以后如何被处理并产出可展示、可导出的结果”展开。

- 技术理解：后端采用 FastAPI + 服务层拆分，把“路由层（接口契约）”与“业务层（文本提取/评分/预览/导出）”分离，降低耦合并便于测试与替换实现；对外部依赖（模型服务）用降级机制控制不确定性。
- 已完成进度：文本提取（PDF/DOCX/纯文本）与安全落盘；登录会话与 token 校验；AI 评分（模型调用 + 规则回退）；预览（PDF 原生、DOCX 转 HTML + 缓存）；评分结果导出（CSV/Excel）。
- 关键实现思想：评分侧坚持“结构化输出”以保证可解析与可落库；模型不可用时通过规则回退保证服务可用；预览侧用统一入口屏蔽格式差异，并通过缓存复用降低重复转换成本；导出侧用内存生成文件避免额外磁盘 IO 并提升响应速度。
- 协作与后续方向：已把爬虫写入与博客判定流程抽象为接口契约与稳定状态输出，便于后续接入真实爬虫/判定服务并持续迭代规则或模型。

## 博客 1（pzl-1.md）摘要

- 主题：后端基础能力建设，围绕“文本提取、文件上传、用户认证”三块把底座搭稳。
- 文本提取策略：按文件类型选择不同提取方式（纯文本直读、PDF 分页提取再拼接、DOCX 读取段落并跳过空段落），统一返回“结果 + 错误原因”，便于上层判断与兜底。
- 文件落盘安全：用随机标识生成唯一文件名避免重名；对用户上传的文件名做安全处理以阻断路径遍历；写入前确保目录存在并做空文件校验。
- 认证链路：学号格式校验；密码使用“盐值 + 强哈希”存储；注册时校验并保存电子签名图片；会话凭据支持生成、校验与过期清理，确保可控。

**专有名词解释**
- PDF / DOCX：常见文档格式；在本项目中用于学生上传申请书/任务书，后端需要对 PDF/DOCX 做文本提取与预览转换以供评分与展示。
- UUID：随机唯一标识；在本项目中用于生成上传文件的唯一文件名/会话 token 的随机源，避免重名冲突并提升安全性。
- basename：从路径里取出文件名的函数；在本项目中用于处理用户上传的原始文件名，防止构造路径导致写入到非预期目录。
- 路径遍历：攻击者通过构造路径访问服务器不应暴露的位置；在本项目中通过 basename 与固定存储目录策略避免覆盖/读取敏感文件。
- 盐值（salt）：与密码一起参与哈希的随机值；在本项目中用于保证相同密码也会生成不同哈希，提升账号密码存储安全。
- PBKDF2：密码派生/哈希算法；在本项目中用于对密码做高成本哈希，增加暴力破解难度。
- token（会话凭据）：登录后颁发给客户端的凭据；在本项目中前端携带 token 调用后端接口，后端据此识别用户与权限。
- token 哈希存储：数据库只存 token 的哈希值；在本项目中用于降低数据库泄露时的风险，避免泄露可直接使用的会话凭据。

## 博客 2（pzl-2.md）摘要

- 主题：后端核心业务能力落地，包括 AI 评分、文件在线预览与评分数据导出。
- AI 评分流程：通过系统提示词明确评分维度与结构化输出；根据模型配置是否可用选择“模型评分”或“规则回退”，并在异常时自动降级保证服务可用。
- 规则回退：用关键词命中等简单特征生成基础分与解释文本，同时对缺失正文/模型失败等情况标记需要人工复核。
- 预览服务：提供统一入口；PDF 走浏览器原生渲染；DOCX 转 HTML 并注入基础样式，同时做缓存判断避免重复转换。
- 数据导出：同时支持 CSV 与 Excel；CSV 通过带 BOM 的 UTF-8 提升 Excel 打开兼容；Excel 通过内存缓冲生成文件减少磁盘 IO。

**专有名词解释**
- Prompt（系统提示词）：给模型的“规则说明”；在本项目中用于约束评分维度与输出字段，保证后端能稳定解析评分结果并写入数据库。
- OpenAI 兼容接口：遵循 OpenAI Chat API 结构的接口规范；在本项目中用于让评分服务可以在不同模型提供方之间切换，降低供应商绑定。
- 结构化 JSON 输出：让模型返回固定字段的 JSON；在本项目中便于后端直接解析 practicality/innovation/total 等分数与理由，用于展示与导出。
- 降级/回退：外部依赖不可用时切换到替代方案；在本项目中用于模型调用失败/未配置时自动改用规则评分，保证评分服务可用。
- HTML 缓存：转换结果复用策略；在本项目中用于 DOCX 转 HTML 预览的缓存复用，减少重复转换消耗并加快预览打开速度。
- BOM（Byte Order Mark）：CSV 文件开头的标记字节；在本项目中用于解决评分结果 CSV 被 Excel 打开时的中文乱码问题。
- openpyxl：Python 读写 Excel（.xlsx）的库；在本项目中用于把评分结果导出为 Excel，便于老师或管理员离线查看与汇总。

## 博客 3（pzl-3-爬虫接口与博客数据写入设计.md）摘要

- 主题：为爬虫协作设计“可稳定入库”的博客模块（模型、接口、鉴权与清洗边界）。
- 数据模型：BlogPost 作为每篇博客的最小记录，用于展示与统计；StudentBlogBinding 作为“学号 → 博客入口地址”的输入源，通常来自任务书上传并支持覆盖更新。
- 写入建议：支持幂等写入与批量写入，便于重复跑爬虫任务、失败重试与提升吞吐；返回成功/失败明细便于排障。
- 鉴权隔离：爬虫调用建议使用独立的 crawler token 或白名单策略，避免复用管理员/用户凭据导致权限泄漏风险。
- 清洗边界：HTML 差异与去噪尽量放在爬虫侧，后端聚焦字段校验、存储、检索与稳定可枚举的 status 输出，供前端统计。

**专有名词解释**
- 爬虫（crawler）：自动从网站抓取内容并结构化处理的程序；在本项目中由后端同学或独立服务抓取学生博客内容并写入 BlogPost。
- Markdown：轻量标记语言；在本项目中用于存储清洗后的博客正文与部分预览内容，便于前端渲染与后端检索。
- 幂等：同一请求重复执行结果一致；在本项目中用于保证爬虫重复抓取或重跑任务时不会产生重复博客记录。
- upsert：更新或插入；在本项目中用于按 url 或 (user, url) 覆盖更新博客内容与状态，便于增量更新。
- 批量写入：一次请求提交多条记录；在本项目中用于爬虫把多篇博客一次性写入，提高吞吐并减少网络往返。
- 白名单：只允许特定来源通过；在本项目中可作为爬虫接口的安全策略，避免被未授权调用。
- status 枚举：用固定字符串集合表示状态；在本项目中用于把博客统一归类为 normal/abnormal，以支持后台红/绿计数统计展示。

## 博客 4（pzl-4-后端-博客正常与不正常判定流程.md）摘要

- 主题：博客质量二分类（normal/abnormal）的后端判定流程，目标是让后台统计稳定、结果可解释、规则可迭代。
- 为什么放后端：涉及特征抽取、模型推理、版本追溯与统一输出，前端不适合承载也难以保证一致性。
- 输入/输出约定：输入包含 url 与清洗后的正文；输出至少包含 status 与 reason，并可扩展置信度分数、标签与规则/模型版本。
- 规则优先基线：先用可解释规则落地（例如代码占比、符号密度、自然语言段落数等）；“科普型”可用项目要素缺失等特征保守标记并记录原因。
- 流水线（pipeline）：抓取清洗 → 特征提取 → 规则判定 →（可选）模型校正 → 写回数据库；短文本与混合型内容是误判高发，需要长度阈值与复核策略。

**专有名词解释**
- 特征抽取：从文本中计算可量化指标；在本项目中用于把博客正文转换成可判定的数值特征（代码占比、段落数等），支撑规则或模型判断。
- 符号密度：代码符号占比指标；在本项目中用于识别“代码粘贴型”博客的候选特征，辅助标记 abnormal。
- reason：判定理由；在本项目中用于后台复核解释，帮助管理员理解红色计数的来源。
- labels（多标签）：更细的原因标签；在本项目中可用于区分 abnormal 的具体类型（例如 code_only / popular_science_only），便于统计与后续优化规则。
- 版本追溯：记录规则/模型版本；在本项目中用于回放同一篇博客在不同规则版本下的判定结果，方便迭代与对比。
- pipeline（流水线）：固定步骤串联处理；在本项目中用于标准化“抓取清洗→特征→判定→写回”的过程，保证结果可复现。

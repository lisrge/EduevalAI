# 前端架构笔记（一）：启动流程、状态边界与访问控制

## 1. 这篇笔记写什么

这篇笔记聚焦“前端工程如何稳定启动并保持可维护”——不讨论某个页面的 UI 细节，而是讨论三个更底层的问题：

- 应用从加载到可交互，哪些初始化必须有明确顺序？
- 哪些信息应当被集中管理，避免在页面之间互相传递与重复判断？
- 用户访问控制应当放在哪里做，才能既统一又易扩展？

## 2. 页面规模与入口规划

随着功能增长，页面会自然分层：

- 用户入口：登录、注册
- 主流程页面：申请书列表、详情与预览、个人空间
- 文档页面：草稿列表与编辑器（面向个人的资料维护）
- 管理页面：后台的权限与数据查看
- 扩展页面：作业收集占位、对话与文件能力入口

把这些入口在路由层组织清晰，后续加新功能时才不会出现“页面找不到归属、权限判断散落各处”的维护问题。

## 3. 启动顺序：把“全局副作用”集中到一个地方

前端最容易出问题的地方是“初始化分散”：主题、路由、全局样式、状态恢复分别写在不同文件里，最终会导致页面首屏闪烁、状态不一致、或首屏请求带着错误的鉴权信息。

本项目的做法是把所有全局初始化收口在入口文件中，并遵循一个原则：

- 先把框架与插件挂载好（状态管理、路由系统）
- 再做与用户体验相关的初始化（例如主题）
- 最后才挂载到 DOM，让用户看到的是最终状态

这样做能最大化减少“首屏渲染后又被二次修正”的情况。

## 4. 根组件最小化：只承载路由出口

根组件承担的职责越少越好。把业务逻辑塞进根组件，长期会造成：

- 全局状态变更难以追踪
- 组件复用困难（根组件变成大杂烩）
- 后续加入布局（例如后台布局）会非常痛苦

因此根组件保持“只负责渲染当前页面”，业务逻辑下沉到具体页面或 store 中。

## 5. 状态边界：登录态是一类“系统级状态”

登录态不是页面私有状态，而是系统级状态。它需要满足：

- 页面刷新后仍能恢复（否则用户体验很差）
- 无效凭据能自动失效并清理（否则会造成持续报错）
- user 信息与 token 的生命周期一致（避免“有 token 没 user”或“有 user 没 token”）

因此前端用一个单独的 store 统一管理“当前是谁、是否已登录、是否具备管理员权限”等信息，并提供一个“启动恢复”入口，确保路由守卫与页面请求都建立在同一套状态判断之上。

## 6. 访问控制：用路由元信息表达规则，用统一入口执行

访问控制有两类：

- 是否必须登录
- 是否必须具备管理员权限（含初始管理员）

如果把这两类判断散落在每个页面里，会出现：

- 有的页面忘了判断，造成越权入口
- 判断逻辑不一致，导致体验割裂

因此路由层用元信息表达规则，统一在一个位置做拦截与跳转策略：

- 未登录访问受限页：跳转到登录，并保留原目标以便登录后回到原处
- 非管理员访问后台页：跳回主流程入口，避免暴露后台页面结构
- 已登录访问登录/注册：直接回到主流程入口

这套规则的价值是：新增页面时只要标记元信息，不需要复制任何判断代码。

## 7. 认证页面：重点是“失败路径清晰”

认证页面常见失败路径包括：格式错误、必填缺失、网络失败、后端拒绝、凭据过期等。前端在认证页面更关注：

- 输入校验要提前做，减少无意义请求
- 错误信息要可读，让用户知道下一步怎么修复
- 提交过程要有明确状态，避免重复点击触发多次请求

这些细节决定了系统“第一印象”，也决定了后续功能是否有人愿意用。

## 8. 小结

这篇笔记总结的是一套“长期可维护”的前端底座：启动顺序清晰、根组件最小化、登录态集中管理、访问控制统一执行。它们不会在演示里最显眼，但会决定后续迭代速度与线上稳定性。

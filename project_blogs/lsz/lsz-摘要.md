# lsz 文件夹博客摘要（1-4）

## 个人工作总结（lsz）

lsz 主要负责项目前端的搭建与主流程页面开发，目标是让用户“登录后能顺畅完成申请书上传、预览与评分相关操作”，同时为管理员提供可管理的后台入口。

- 技术理解：用 Vue3 的组件化拆分页面职责，用 Pinia 把登录态/列表态等系统级状态集中管理，避免状态在页面之间层层传递；用 Vue Router 的 meta + 全局守卫把鉴权逻辑收口，保证权限规则一致且可扩展。
- 已完成进度：登录/注册与登录态恢复；申请书工作台（列表、上传、预览面板、评分入口）；“我的申请书/任务书”草稿与编辑器；个人空间；后台管理入口与数据管理表格页面。
- 关键实现思想：批量上传采用“队列 + 有限并发”，把不可控的多文件并行变成可推进的流程，并用状态机把“排队/进行中/成功/失败”清晰回写到 UI，提升稳定性与可解释性。
- 可扩展准备：预留聊天与拖拽上传入口，为后续接入智能体能力与更复杂交互留出前端结构空间。

## 博客 1（lsz-1.md）摘要

- 交代项目背景与分工：项目面向“实训过程跟踪”，首期目标是把申请书接收与自动评分的最小流程跑通。
- 功能覆盖：登录/注册、申请书管理入口、个人文档列表、个人空间等基础页面。
- 工程入口要点：入口文件中先注册状态管理与路由系统，再在挂载前初始化主题与全局样式，减少首屏闪烁。
- 路由策略：页面组件采用懒加载减少首屏体积；用路由守卫统一处理“需登录页/游客页”的访问规则与跳转。
- 登录体验：补充学号格式校验、必填校验、错误提示、自动登录选项等，使失败路径更可解释。

**专有名词解释**
- Pinia：Vue 生态的状态管理库；在本项目中用于集中管理登录态与用户信息，以及申请书列表等跨页面状态，避免每个页面重复拉取与判断。
- Router（Vue Router）：前端路由系统，决定 URL 与页面组件的映射关系；在本项目中用于组织登录/注册、申请书、个人空间、后台管理等入口，并承载统一的访问控制策略。
- app.mount：把 Vue 应用挂载到页面 DOM 节点；在本项目中用于控制“主题与全局样式初始化完成后再渲染”，减少首屏闪烁与二次修正。
- 懒加载（lazy load）：页面在访问到时才加载对应代码；在本项目中用于降低首屏体积，让登录后进入申请书页面的加载更快。
- beforeEach（路由守卫）：在每次路由跳转前执行的统一拦截逻辑；在本项目中用于集中处理未登录跳转登录页、已登录禁止访问登录/注册页、后台页面权限拦截等规则。
- requiresAuth / guestOnly：路由元信息（meta）标记；在本项目中用于声明页面访问要求（必须登录/仅游客），从而让路由守卫能统一执行跳转逻辑。

## 博客 2（lsz-2.md）摘要

- 先做模块清单：按“统一头部、列表/上传/预览、文档中心、路由与 API、主题样式”等拆分前端结构，方便定位职责。
- 批量上传核心：用队列驱动上传任务，每个文件先生成本地唯一标识并入队，通过队列调度控制并发数量，降低卡顿与失败率。
- 状态呈现：为每个条目设计上传/评分状态，使 UI 能明确显示“排队/进行中/成功/失败”等阶段。
- 评分交互：支持单项评分与批量评分；批量评分一次提交多个目标，按后端结果逐项回写分数与状态。
- 扩展交互：补充了多会话聊天的基本结构，并引入拖拽上传与预览的交互雏形，便于后续接入更多智能体能力。

**专有名词解释**
- queue（队列）：按顺序存放待处理任务的数据结构；在本项目中用于把多文件上传变成“可控推进”的流程，避免一次性并发导致浏览器卡顿。
- 并发上限：限制同时进行的任务数；在本项目中用于控制同时上传/处理的文件数量（例如最多 2 个），提升稳定性与成功率。
- localId（本地唯一标识）：前端为每个文件生成的临时 ID；在本项目中用于在后端返回 applicationId 之前稳定追踪条目状态（排队/上传中/失败等）。
- pumpQueue：队列推进函数；在本项目中负责从 queue 取出待上传任务并启动处理，直到达到并发上限。
- 状态机：用一组有限状态表示流程阶段；在本项目中用于统一描述上传与评分阶段，保证列表与详情面板展示一致。
- 批量评分：一次对多个申请书发起评分请求；在本项目中用于管理员对多份申请书集中触发评分，并按后端返回逐项回写结果。
- 拖拽事件（dragenter/dragover/drop）：浏览器拖拽相关事件；在本项目中用于实现聊天窗口的拖拽上传入口，支持快速选择文件并生成预览。

## 博客 3（lsz-3-工程骨架与认证闭环.md）摘要

- 聚焦前端底座三件事：启动顺序、状态边界、访问控制，强调这些比单页 UI 更决定长期维护成本。
- 启动顺序：全局初始化集中在入口，并遵循“先挂载框架能力 → 再做体验初始化 → 最后渲染”的顺序，减少首屏二次修正。
- 根组件最小化：根组件只承载路由出口，业务逻辑下沉，避免全局耦合扩大。
- 系统级状态：登录态需要统一管理并提供启动恢复入口，确保路由拦截与接口请求共享同一套判断。
- 访问控制：用路由元信息表达规则，并在统一入口执行拦截/跳转，新页面只需标记规则。
- 认证页面原则：把失败路径讲清楚（输入校验、错误可读、提交状态明确），提升可用性。

**专有名词解释**
- 根组件：应用最外层组件；在本项目中保持“只做路由出口”，避免把申请书/后台等业务逻辑塞进根组件导致耦合。
- 系统级状态：影响整个应用的状态（例如登录态、用户角色）；在本项目中用于驱动路由拦截与页面功能展示（普通用户/管理员看到不同操作）。
- 访问控制：对页面/接口进行权限限制；在本项目中用于保护后台管理入口与敏感操作，保证普通用户无法进入管理员页面。
- 路由元信息（route meta）：附加在路由配置上的额外字段；在本项目中用于标记页面是否需要登录、是否需要管理员权限，从而让统一守卫执行规则。
- 重定向（redirect）：把用户从当前地址跳转到另一个地址；在本项目中用于未登录访问受限页面时跳转登录，并在登录后回到原目标页面。

## 博客 4（lsz-4-申请书工作台与文档中心.md）摘要

- 双通道设计：把“频繁编辑”和“后端处理/归档”拆开，分别对应草稿通道（结构化内容、可高频保存）与归档通道（统一上传触发后端抽取/预览/评分）。
- 分层承载复杂度：页面做入口与布局；组件做展示与交互；store 管队列/并发/重试/状态回写；API 层统一网络细节。
- 队列推进：用少量并发 + 排队推进降低不确定性，同时让错误可解释、可回写。
- 评分可追踪：支持多选一次提交的批处理方式；前端聚焦“收集目标 + 回写结果”，评分逻辑由后端执行。
- 管理端诉求：后台以“快速定位”为目标，提供权限与数据两类入口，并用统一访问控制保护后台路由。

**专有名词解释**
- 草稿通道：用于编辑与保存文档内容的通道；在本项目中对应“我的申请书/任务书”可反复编辑与自动保存的能力，提升编辑体验。
- 归档通道：用于提交最终文件给后端处理的通道；在本项目中用于触发后端的文本抽取/预览/评分流程，让提交后的材料进入统一管理与统计体系。
- store：状态管理层（例如 Pinia store）；在本项目中用于承载上传队列、选中条目、加载状态、错误信息等流程状态，避免组件间互相传参。
- 重试：请求失败后再次尝试的策略；在本项目中可用于上传/评分失败后的再次提交，降低临时网络波动带来的影响。
- 回写：把后端返回的结果更新回前端条目；在本项目中用于把上传结果、评分分数、失败原因写回列表与详情面板。
- 批处理：将多个目标合并成一次请求或一次处理；在本项目中体现为批量上传与批量评分，提升操作效率并减少请求次数。
